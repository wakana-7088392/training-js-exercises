# ch15.11-15/ex01

## 問題

### Slack や O365 の API は機密情報をヘッダーで指定する。パスやクエリでないのは何故かセキュリティの観点で記載しなさい。

- Copilotの回答

- URLの露出を防ぐ
    - パスやクエリパラメータに機密情報を含めるとURLがログやブラウザの履歴に残る可能性があり、それがセキュリティリスクとなる。ヘッダーに含めることでリスクを軽減可能。
- キャッシュの問題
    - URLに機密情報を含めるとプロキシサーバやブラウザのキャッシュに保存される可能性がある。これにより機密情報が意図せず保存され、第三者にアクセスされるリスクが高まる。
- ヘッダーの暗号化
    - HTTPヘッダーがリクエストの一部として暗号化されるため、より安全にデータを送信できる。
- アクセス制御の強化
    - ヘッダーを使用することでAPIトークンや認証情報を簡単に管理・更新でき、アクセス制御を強化することができる。

- 調べた回答
    - ヘッダーのほうがセキュリティ保護を付与することができ、安全性が高いから。
        - そもそもヘッダーには「Content-Security-Policy」などを始めとしたセキュリティヘッダーが存在する。
        - Slackにはx-slack-signatureというHTTPヘッダーがあり、アプリに送信するすべてのリクエストに付与されている。これにはHMAC-SHA256キーつきハッシュが設定されるため、漏洩するリスクが減る…のではないか。
